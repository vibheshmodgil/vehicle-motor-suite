§0Method & scope

This review covers the whole repository — every module under vmi/, the test suite, dependency posture, and repo hygiene — not just the physics/report work from the last few sessions. Findings are ranked by what actually happens if left alone, not by how they were found.

Method: an AST-level duplicate-definition scan (catches methods silently shadowed within a class, and name collisions across the mixins that make up TorqueSpeedApp), a pyflakes pass across every module, a manual re-read of the battery-limit solver, range-energy model, and the new report-interpretation engine added this session, and a file-system/git-hygiene check (CWD-dependent paths, .gitignore coverage, CI presence). The existing 181-test golden-value suite was run before and after every change.

§1Findings register

F-01 High Open

Session, log, and motor-library files resolve against the current working directory, not against the app's install location

vmi/app.py:61 · vmi/data_io.py:78,88 · vmi/applog.py:13 · vmi/enhancements.py:1506

std_motor_data_sample.json, vmi_app.log, and vmi_last_session.json are all opened with a bare relative filename. Python resolves that against the process's current working directory at launch — not "next to main.py," which is what the docstrings in main.py and applog.py both claim. Any launch path that isn't "open a terminal, cd into this exact folder, run python main.py" — a desktop shortcut, a different terminal tab, python -m vmi invoked from elsewhere — silently creates fresh, empty versions of all three files in whatever directory happened to be current, instead of finding the real ones. The saved motor library "disappears," the session doesn't restore, and errors stop being logged where anyone would look.

Recommendation Anchor all three paths to a fixed location — Path(__file__).resolve().parent.parent from inside the vmi package gives "next to main.py" regardless of CWD. This is a config change, not a rewrite; low risk, high support-cost payoff.
F-02 Critical Fixed this pass

Dead method with two undefined names and two hardcoded, nonexistent input files

vmi/drive_cycle.py:23 (removed)

DriveCycleMixin.drive_cycle_efficiency() read "check.xlsx" and "idc.xlsx" directly off disk — bypassing the app's entire upload/column-picker pipeline — and called potting_efficieny(...) and driveCycleData(...), neither of which exist anywhere in the codebase. Confirmed via full-repo grep: the method is never called from any button, menu, or dispatch branch. It is a prototype fragment that survived the "copy every method body verbatim" migration described in CLAUDE.md. Harmless as long as it stays uninvoked — but indistinguishable from live code to a future reader, and one accidental wiring away from an immediate NameError crash.

Resolution Removed. Confirmed zero callers before deletion; full test suite (181/181) and a package import check both pass afterward.
F-03 High Fixed this pass

Report-picker logic was defined twice in the same class — the first copy silently dead

vmi/enhancements.py:333–386 (removed), :1129 (kept)

EnhancementsMixin.generate_report() and ._open_report_picker() each had two method bodies in the class definition — a shorter, older pair and a newer pair with the "full view hierarchy" picker text. Python keeps only the last definition of a given name in a class body, so the app has been correctly running the second copy all along — this was never a runtime bug. It was, however, ~55 lines of plausible-looking dead code sitting in the same file as the real thing, with no marker distinguishing them. The concrete risk: a future edit lands in the shadowed copy, the developer sees no effect, and concludes the change "doesn't work" or — worse — the two copies are later merged in a way that resurrects the older, less complete behavior.

Resolution Removed the shadowed first copy. Verified the second (kept) copy's picker text and behavior are what the app actually renders; 181/181 tests plus a full report-generation smoke test pass afterward.
F-04 Medium Open

57% of the codebase — including the entire report-interpretation engine — has no automated test coverage

vmi/dispatch.py · torque_force.py · drive_cycle.py · engine.py · compare_std.py · enhancements.py · graph_settings.py · report_insights.py — 8,212 of 14,510 lines in vmi/

The 181-test suite is genuinely strong where it exists — golden-value regression tests lock physics.py, calc_ext.py, units.py, mtpa_mtpv.py's solver, mechanical_design.py's formulas, and bom.py's tree math. But it structurally cannot see widget wiring, dispatch routing, Excel/JSON I/O, or plot orchestration — and it did not, and could not, catch F-02 or F-03. Both surfaced in under five seconds from a single pyflakes pass with zero project-specific setup. The 956-line report_insights.py engine added this session (per-view engineering interpretation for the HTML report) is presently exercised only by ad hoc scripts run by hand during development, not by anything committed to tests/.

Recommendation Two independent, low-cost moves: (1) add pyflakes (or ruff) as a standing pre-commit/CI gate — it needs no project-specific config and already found two real bugs; (2) commit a lightweight smoke suite that boots TorqueSpeedApp off-screen (withdraw()), clicks through every analysis type, and generates one full report — this exact pattern was used by hand throughout this session and is roughly 150 lines to make permanent. It won't catch numeric regressions, but it will catch every crash-on-launch and dead-reference class of bug going forward.
F-05 Low Open

An entire input section is built but never shown — pre-dates the Torque/Force merge

vmi/app.py:370–374 (sections['simforce']) · vmi/dispatch.py:189–190

The "Simulation Settings-force" section and its two entries (xlim_force, ylim_force) are constructed in __init__ but never appear in any analysis_sections list, so the user can never see or edit them — CLAUDE.md itself documents that Torque and Force were merged into one "Powertrain Sizing" analysis using a different, shared set of limit fields. dispatch.py still calls get_x_limits_force()/get_y_limits_force() against the orphaned fields every single plot and never uses either result.

Recommendation Safe to delete: the widgets, the two dead locals in dispatch.py, and (if nothing else references them) the two limit-getter methods in limits.py. Left open in this pass — touches the widget tree and is zero-urgency, but should not be mistaken for a hidden feature by a future maintainer.
F-06 Low Open

Assorted unused locals — lint noise, no behavior change

compare_std.py · dispatch.py · efficiency.py · torque_force.py · data_io.py (~12 sites)

Mostly the pattern from .physics import df at module scope, shadowed by a local df = ... inside nearly every method (harmless — the import is never used by name after that point) — plus a couple of genuinely-computed-then-discarded values in compare_std.py (speed_km_hr_wheel, force_std) that are correctly recomputed a few lines later under the actual branch that needs them. No fix applied to individual sites here; the pyflakes gate recommended in F-04 keeps this class of thing from silently accumulating without hunting each one by hand.

§2Architecture — what the mixin model costs

TorqueSpeedApp is assembled from thirteen mixins sharing one self with no encapsulation between them — documented deliberately in CLAUDE.md as a tradeoff, and it is a reasonable one for a single-developer engineering tool: it keeps thirteen files simple at the cost of one systemic risk. That risk is real and specific, not hypothetical: a renamed self.* attribute is a cross-file change with no import graph to catch it, and two mixins defining the same method name means the earlier one in the MRO silently wins with no error.

The AST scan run for this audit checked the second failure mode directly — it walks every class in every module and flags any method name defined in more than one mixin. Result today: zero collisions. That is a genuine, current strength, not a gap — but it is presently maintained by discipline (CLAUDE.md's "grep before naming a new method") rather than by anything that runs automatically.

The twenty-line AST scan used to verify this is cheap enough to keep. Wiring it into the F-04 lint gate turns "currently zero collisions" into "structurally cannot regress" for the cost of one more check in the same CI step.

Separately: 194 occurrences of except Exception: pass across vmi/ are a deliberate fail-soft choice, appropriate for a GUI where one bad field shouldn't kill the session — but several of them now wrap load-bearing calculation paths added this session: the battery-efficiency map lookup (_battery_eta_fn) and every builder function inside report_insights.py. A genuine defect in either currently degrades silently — to a constant-efficiency fallback, or to a report section that simply carries no interpretation — with no trace anywhere. This is precisely the failure mode that let F-02 go unnoticed. Recommend narrowing scope only where it's cheap: log to vmi_app.log on the exception before swallowing it, in the calculation paths specifically (not the GUI event handlers, where the existing pattern is correct as-is).

§3What's already solid

An audit that only lists problems is an incomplete one. Several things checked out clean and are worth stating plainly rather than leaving implicit:

  • No dangerous APIs anywhere in vmi/ — a full-repo grep for eval, exec, os.system, subprocess, and pickle returned nothing. For a tool that loads user-supplied Excel/JSON files, that absence is a real property, not a coincidence.
  • The local-LLM assistant sidebar is actually local — the only network endpoint referenced anywhere in llm_client.py/rag_store.py/assistant.py is http://localhost:11434 (Ollama). The "no cloud calls" claim in the code comments holds up.
  • .gitignore hygiene is correct — generated state (vmi_last_session.json, vmi_app.log*, the user's own std_motor_data_sample.json, the assistant's vector index, sample data) is excluded and confirmed untracked; licensed reference material dropped into knowledge_base/ is excluded by default with an explicit opt-in comment.
  • Dependencies are exactly pinned, not range-pinned, with a comment explaining why (customtkinter's breaking changes between versions) — the right call for a desktop tool where "it worked when I tested it" needs to keep meaning that.
  • The physics core is genuinely golden-locked — physics.py, the battery/inertia helpers, the MTPA/MTPV solver, and the mechanical-design formulas all have regression tests tied to specific numeric literals, exactly matching the project's own stated contract that calibrated numbers must not silently drift.
